# 一、输入参数规范（Web界面映射）

## 1.1 基础参数（用户输入）
| 参数名 | 类型 | 取值范围 | 单位 | 说明 |
|--------|------|---------|------|------|
| 建筑长度 | float | 8.0 ~ 16.0 | m | 开间方向，建议模数0.3m |
| 建筑宽度 | float | 8.0 ~ 14.0 | m | 进深方向，建议模数0.3m |
| 楼层数 | int | 1 ~ 4 | 层 | 农房不宜超过4层 |
| 层高 | float | 3.0 ~ 3.9 | m | 首层可3.6~3.9m，标准层3.0~3.3m |
| 屋顶类型 | enum | 双坡屋顶/平屋顶/四坡屋顶 | - | 影响顶层轮廓 |
| 建筑风格 | enum | 现代简约/传统民居/徽派/岭南 | - | 仅影响立面与材质，不影响平面规则 |
| 开窗密度 | enum | 低/中/高 | - | 软约束，影响外窗生成 |

## 1.2 功能区数量输入（各层独立设置）
用户为每层指定以下功能区的数量（0-N）：
- 客厅、餐厅、厨房、卧室、卫生间、书房、露台、储藏室、楼梯间、门廊

**注意**：楼梯间数量必须为1，且所有楼层强制存在。

# 二、硬约束规则（Hard Constraints）
**说明**：以下规则必须100%满足，否则视为非法平面。代码生成后需逐项自检。

## 2.1 垂直对齐规则（核心强制）

### R1. 楼梯间全层锁定
- **规则**：所有楼层的楼梯间必须在同一平面位置，长、宽尺寸完全一致（误差0cm）。
- **理由**：结构贯通、疏散安全、施工可行性。
- **代码实现**：首层确定楼梯间坐标(x,y,w,h)后，上层直接复用该Rect。
- **3D联动**：楼梯间竖向贯通，生成统一楼梯井体块。

### R2. 卫生间垂直对齐
- **规则**：二层及以上所有卫生间（含主卫、公卫）的平面投影必须**完全落入**一层卫生间轮廓内。
- **允许**：上层卫生间面积 ≤ 下层，形状可微调（如L型变矩形）。
- **禁止**：上层卫生间相对下层发生平移、扩大或超出边界。
- **例外**：若一层无卫生间（理论上不允许，见R7），则二层卫生间需与一层厨房管井对齐。

### R3. 承重墙竖向连续
- **规则**：楼梯间、卫生间周边的承重墙体，上下层必须对齐。
- **禁止**：上层承重墙压在下层非承重区域或开门洞上方。

## 2.2 空间满铺与重叠规则（强制）

### R4. 面积100%功能化
- **规则**：每层建筑面积（外墙内围净面积）必须被所有功能区完全铺满，不得出现未标注的空白区域。
- **允许例外**：仅允许存在 ≤ 2㎡ 的管道井、风道或结构加强区，且必须贴临卫生间或厨房，需在图中标注"管井"。
- **禁止**：无功能的过道 > 1.5m；若存在长走廊，必须将其并入相邻功能区（如客厅延伸区、过道厅）。

### R4a. 自适应面积填充（v3 新增）
- **规则**：当房间目标总面积 ≠ 可用净面积时，必须执行自适应填充：
  - **总面积 < 可用面积**：优先按面积比例放大客厅、主卧、餐厅；其次放大次卧；最后放大厨房、卫生间（不得超过各自 maxArea 上限）。
  - **总面积 > 可用面积**：优先压缩储藏室、书房（可取消）；其次压缩次卧面积；再次压缩客厅进深；厨房、卫生间、楼梯间严禁压缩至低于硬约束最小值。
  - **计算基准**：所有面积以净面积（使用面积）计算，已扣除墙体厚度。
- **代码实现**：在 `allocateRoomAreas` 之后增加二次平衡（rebalance）步骤。

### R4b. 禁止空间重叠（v3 新增）
- **规则**：同一楼层内，任意两个功能区的矩形投影不得重叠。
- **特别禁止**：上层露台/阳台的平面投影不得与楼梯间、客厅/起居厅的投影重叠。露台必须是建筑轮廓的**外挑或贴边**区域，不能与核心筒冲突。

### R5. 墙体厚度扣除
- **规则**：计算功能区净面积时，必须扣除墙体厚度（外墙240mm，内墙120mm或200mm）。
- **面积基准**：所有面积标准均指**净面积**（使用面积）。

## 2.3 核心流线规则（强制）

### R6. 楼梯间出口锁定客厅
- **规则**：所有楼层，楼梯间出口（走出楼梯间的第一区域）必须是客厅（或门厅/堂屋）。
- **禁止**：楼梯间直接连接卧室、卫生间、厨房、书房、露台等其他区域。
- **缓冲要求**：若楼梯间直接对外开门，门口必须设 ≥ 1.5㎡ 的缓冲区域（门斗/玄关），该缓冲属于客厅或门厅的一部分。

### R6a. 卫生间独立动线（v3 新增，修复套流漏洞）
- **规则**：卫生间**不得**只能通过厨房或餐厅进入。
- **必须满足以下至少一项**：
  1. 卫生间有一扇门直接通向公共区域（客厅、门厅、过道）；
  2. 卫生间位于卧室区走廊尽头，且走廊本身可从客厅直接到达（不穿越厨房）；
  3. 若卫生间位于厨房同侧，则必须保证卫生间与客厅/餐厅之间存在不穿越厨房的邻接边，且在该邻接边上开设门洞。
- **禁止的套流模式**：
  - ❌ 客厅 → 厨房 → 卫生间（必须通过厨房）
  - ❌ 客厅 → 餐厅 → 厨房 → 卫生间（必须通过厨房）
  - ❌ 卧室 → 卫生间（单门独卫不在此限，但公卫不允许）
- **代码实现建议**：在生成布局后，使用 BFS 从客厅出发遍历到卫生间，检查路径中是否必须经过厨房或餐厅；若是，则交换卫生间与相邻卧室/储藏室的位置，或重新分割后带区域。

### R7. 客厅作为交通枢纽
- **规则**：从客厅必须能够直接通达该层所有其他功能区（卧室、厨房、餐厅、卫生间、书房、露台）。
- **禁止**：出现"穿过卧室去卫生间"或"穿过厨房去餐厅"的套间式流线。
- **门向规则**：所有房间的门应面向客厅、过道或公共区域开启；禁止两个私密空间直接对门（如卧室门正对卧室门）。

### R7a. 过道（走廊）合法化（v3 新增）
- **规则**：当建筑进深较大（≥11m）或房间数量较多时，允许设置**内走廊**作为交通空间。
- **过道参数**：
  - 净宽：0.9 ~ 1.2m（单人通行下限 0.9m，双人交错 1.2m）。
  - 若过道净宽 > 1.5m，则超过部分必须并入相邻功能区（客厅、餐厅等），不得作为纯交通空间浪费。
  - 过道必须有自然采光或人工照明标注，禁止设置封闭暗走廊。
- **过道与R4的关系**：过道面积计入使用面积，必须包含在面积分配中，不得作为未标注空白。

## 2.4 功能区存在性规则（强制）

### R8. 一层必备功能区
一层必须包含且仅包含以下功能区（数量≥1）：
- **楼梯间**：1个
- **客厅/堂屋**：1个，净面积 ≥ 18㎡
- **厨房**：1个，净面积 ≥ 6㎡，必须贴临外墙
- **餐厅**：1个，净面积 ≥ 6㎡（可与客厅合并为客餐厅，但需明确分区标注）
- **卧室**：≥1间，其中至少1间为老人房（优先朝南），净面积 ≥ 9㎡
- **卫生间**：≥1间（公卫），净面积 ≥ 4㎡，优先贴临卧室区域

### R9. 二层及以上必备功能区
二层及以上必须包含：
- **楼梯间**：1个（与一层完全对齐）
- **客厅/家庭厅**：1个，净面积 ≥ 15㎡，必须位于楼梯间出口正接区域
- **卧室**：≥2间，其中1间为主卧（净面积 ≥ 12㎡，优先朝南）
- **卫生间**：≥1间（公卫，与一层竖向对齐），若设主卫则主卫也必须与下层对齐
- **露台/阳台**：≥1个，净面积 ≥ 6㎡，必须设置防护栏杆（平面图中标注栏杆线）

### R10. 三层及以上模式
三层及以上功能应与二层保持一致，可选两种模式：
- **模式A（客房模式）**：2~3间卧室 + 1公卫 + 1小起居室（≥10㎡，替代大客厅）
- **模式B（套间模式）**：1主卧套间（卧室≥15㎡+独卫+衣帽间≥6㎡）+ 1~2次卧 + 1公卫 + 家庭厅（≥12㎡）

## 2.5 几何尺寸红线（强制）

### R11. 房间最小净尺寸
| 功能区 | 最小净宽 | 最小净深 | 最小净面积 | 长宽比上限 |
|--------|---------|---------|-----------|-----------|
| 客厅/堂屋 | 3.9m | 4.5m | 18㎡ | 1:3 |
| 主卧 | 3.3m | 4.2m | 12㎡ | 1:3 |
| 次卧/老人房 | 3.0m | 3.6m | 9㎡ | 1:3 |
| 厨房 | 2.1m | 2.8m | 6㎡ | 1:3 |
| 餐厅 | 2.4m | 2.5m | 6㎡ | 1:3 |
| 卫生间（公卫） | 1.8m | 2.2m | 4㎡ | 1:2.5 |
| 卫生间（主卫） | 2.0m | 2.5m | 5㎡ | 1:2.5 |
| 书房 | 2.4m | 2.5m | 6㎡ | 1:3 |
| 储藏室 | 1.8m | 2.0m | 3㎡ | 1:3 |
| **楼梯间** | **2.4m** | **3.6m** | **4.5㎡** | **1:2** |
| 露台 | 2.0m | 3.0m | 6㎡ | 1:4 |
| 过道/走廊 | 0.9m | — | — | 1:5 |

### R12. 楼梯间详细规格（v3 修正）
- **最小外框尺寸**：双跑楼梯最小外框 **2.4m × 3.6m**（净宽×净深），严禁低于此值。
- **面积计算**：楼梯间面积 = 开间 × 进深，**不得**用固定值（如 9.5㎡）硬编码。应在满足最小尺寸的前提下，根据楼层布局和搬运需求自然形成面积。
- **梯段净宽**：≥ 1.1m（农房搬运需求，适当放宽）
- **踏步高**：150 ~ 175mm
- **踏步宽**：260 ~ 300mm
- **休息平台**：深度 ≥ 梯段宽度，转向平台 ≥ 1.5m × 1.5m
- **生成策略**：
  - 若建筑长度 ≥ 10m，推荐双跑楼梯外框 2.6m × 4.0m ~ 3.0m × 4.5m。
  - 若建筑长度 < 10m，允许三跑楼梯外框 3.0m × 3.0m（正方形核心筒）。
  - **禁止**：为追求"紧凑"而将楼梯间压缩至 2.4m 以下开间，这会导致梯段净宽 < 1.0m，不符合农房使用需求。

### R13. 门窗洞口底线
- **入户门**：≥ 1.2m，优先朝南/朝东，禁止正对楼梯间或卫生间
- **卧室门**：≥ 0.9m，内开，禁止开在床头墙正中
- **卫生间门**：≥ 0.8m，外开或推拉，禁止正对客厅、餐厅、厨房门
- **厨房门**：≥ 0.9m，可开放（无门洞 ≥ 1.5m）
- **客厅通露台门**：≥ 1.5m，优先落地玻璃推拉门
- **外窗**：可开启面积 ≥ 外窗面积的25%，卫生间必须有外窗或通风井

### R14. 干湿分区与视线遮挡（v3 新增）
- **规则**：卫生间门不得直接面向餐厅座位区、厨房操作台面开启。
- **视觉遮挡**：若卫生间门不可避免地位于客厅视线范围内，必须在门前设置 ≥ 0.6m 深的转折墙或屏风区（可并入卫生间或过道面积）。
- **洁污分离**：卫生间、厨房等湿区与卧室、客厅等干区之间应有明确分界，不宜直接对门。

# 三、软约束规则（Soft Constraints）
**说明**：以下规则为优化目标，在硬约束满足的前提下尽量满足。可用于生成后的评分或迭代优化。

## 3.1 朝向与采光优先级
**朝向优先级排序**：正南 > 东南 > 东 > 西南 > 东北 > 西 > 西北 > 北

| 功能区 | 优先朝向 | 最低采光标准 |
|--------|---------|-------------|
| 一层客厅/堂屋 | 南/东 | 直接外窗，窗地比 ≥ 1/6 |
| 老人房（一层） | 南/东南 | 冬季满窗日照 ≥ 2小时 |
| 二层主卧 | 南/东 | 必须有直接外窗 |
| 厨房 | 东/南（次选北） | 直接外窗，自然通风 |
| 卫生间 | 北/西 | 优先外窗；无外窗时必须设通风井 |
| 楼梯间 | 南/东/西 | 至少一侧外墙，用于采光和疏散 |
| 餐厅 | 南/东 | 可与客厅共享采光或独立外窗 |
| 书房 | 北/东 | 避免西晒，需要稳定采光 |

## 3.2 空间邻接偏好（Adjacency）

### 必须相邻（Positive）
| 空间A | 必须紧邻 | 权重 |
|-------|---------|------|
| 厨房 | 餐厅、外墙 | 10 |
| 卫生间 | 至少1间卧室、外墙/管井 | 10 |
| 楼梯间 | 客厅、外墙（至少一侧） | 10 |
| 老人房 | 一层卫生间、客厅 | 9 |
| 主卧 | 主卫（如有）、露台 | 8 |

### 禁止相邻（Negative）
| 空间A | 禁止紧邻 | 权重 |
|-------|---------|------|
| 厨房 | 卧室床头墙（隔声差） | 10 |
| 卫生间 | 餐厅、厨房（正对门） | 10 |
| 楼梯间下方 | 卧室床头、餐厅座位区 | 8 |
| 客厅 | 农具间/储藏室直接开门 | 7 |

## 3.3 视线与隐私
- **客厅**可见厨房入口、餐厅，但**不应直视**卫生间门（需设转折或遮挡）
- **卧室门**不应正对入户大门或楼梯口
- **卫生间门**必须设视觉转折，禁止从客厅直接看到便器区域

## 3.4 通风与流线优化
- **穿堂风**：客厅/楼梯间宜能接触两个不同朝向的表面，形成通风路径
- **厨房油烟**：必须直排室外，禁止排入楼梯间或室内过道
- **送餐流线**：从厨房到餐厅的流线不应穿越客厅中央（可沿边缘）
- **洁污分离**：卫生间、厨房等湿区与卧室、客厅等干区适当分离

## 3.5 农房特殊场景
- **堂屋/神龛**：如用户选择"传统民居"风格，一层北墙或东墙预留 ≥ 1.5m 宽神龛位，该墙面禁止开门窗，堂屋区域进深 ≥ 3.6m
- **农具/杂物**：储藏室/农具间必须有独立对外出入口或通过北侧次要走廊进入，禁止穿越客厅
- **车库**：若设置，占北侧或西侧，不得占用南向采光面
- **露台下方**：禁止布置厨房或卫生间（防水与管道问题），优先是客厅、餐厅或次要卧室
- **加建预留**：三层轮廓宜与二层对齐或向内退台，禁止向外悬挑 > 1.2m；屋顶楼梯间出口预留上屋顶通道

# 六、生成算法建议（供Kimi Code参考，v3 重大更新）

## 6.1 生成顺序（Pipeline）
1. **接收参数**：长宽、层数、层高、各层功能区数量
2. **建立网格**：以300mm为模数建立建筑网格（如12m×10m → 40×33格）
3. **构建气泡图（Bubble Diagram）**：
   - 将各功能区作为节点，邻接关系作为边（客厅为图中心，度最大）。
   - 必要边：客厅-楼梯间、客厅-餐厅、客厅-卧室、厨房-餐厅、卫生间-卧室（至少一间）。
   - 禁止边（不应直接相邻）：卫生间门-餐厅座位区、厨房灶台-卧室床头。
4. **锁定核心**：
   - 首层确定楼梯间位置（建议北侧或居中偏北），尺寸严格满足 R12（≥2.4m×3.6m）。
   - 首层确定卫生间位置（紧邻楼梯间或北侧，作为竖向基准）。
5. **矩形分割（Rectangular Dissection）**：
   - 采用**切片树（Slicing Tree）**或** Guillotine 分割**策略，保证每个子区域都是矩形且无重叠。
   - 按气泡图的邻接优先级决定分割轴（水平/垂直）和顺序。
6. **面积自适应填充**：执行 R4a 规则，二次平衡各房间面积至总和 = 可用净面积。
7. **门洞路由**：在相邻房间共享边上按门洞宽度（0.8~1.5m）扣除墙体，生成门洞位置。优先保证 R6a 的卫生间独立动线。
8. **环境校验**：检查朝向分配、采光面、视线隐私、R6a 动线。
9. **输出平面**：生成带标注的2D平面（房间名+面积+门窗+栏杆+尺寸线）。
10. **3D体块联动**：根据2D轮廓+层高生成3D白模，楼梯间竖向贯通，露台设栏杆体块。

## 6.2 随机性控制
- **允许随机**：可选功能区的有无、房间具体长宽（在范围内）、门洞位置（在允许墙段内）、非承重墙布局
- **禁止随机**：楼梯间位置尺寸、卫生间位置（相对一层）、必备功能区存在性、硬约束尺寸红线

## 6.3 冲突解决策略
当面积不足无法同时满足所有软约束时，按以下优先级舍弃：
1. 保留硬约束（R1~R14）
2. 保留必备功能区（R8~R9）
3. 优先满足老人房朝南、厨房贴外墙、卫生间有独立动线（R6a）
4. 其次满足书房/衣帽间等可选功能
5. 最后满足理想面积（允许降至最小值）

## 6.4 回溯与局部重排（v3 新增）
- **问题场景**：当按默认顺序生成的布局导致 R6a（卫生间套流）、R4（未填满）或 R4b（重叠）违规时，不应直接输出警告，而应触发局部重排。
- **重排策略**：
  1. 记录违规类型（如"卫生间只能通过厨房到达"）。
  2. 在气泡图中交换卫生间与相邻房间（如储藏室、卧室）的节点位置。
  3. 重新执行矩形分割（限制只重排后带或受影响区域）。
  4. 若交换 3 次仍无解，才输出违规警告并给出次优方案。
- **代码提示**：可使用带迭代次数上限的 while 循环实现，避免无限回溯。

## 6.5 面积自适应填充伪代码（v3 新增）
```javascript
function rebalanceAreas(rooms, availableNetArea) {
  const total = rooms.reduce((s, r) => s + r.targetArea, 0);
  const gap = availableNetArea - total;
  
  if (Math.abs(gap) < 0.5) return rooms; // 已平衡
  
  if (gap > 0) {
    // 面积不足填满：优先放大客厅、主卧、餐厅
    const expandPriority = ['living_room', 'bedroom', 'dining', 'kitchen', 'bathroom', 'study', 'storage'];
    let remaining = gap;
    for (const type of expandPriority) {
      const candidates = rooms.filter(r => r.type === type && r.targetArea < r.maxArea);
      candidates.forEach(r => {
        const headroom = r.maxArea - r.targetArea;
        const add = Math.min(headroom, remaining / candidates.length);
        r.targetArea += add;
        remaining -= add;
      });
      if (remaining <= 0.1) break;
    }
  } else {
    // 面积超限：优先压缩储藏室、书房，再压缩次卧、客厅
    const shrinkPriority = ['storage', 'study', 'bedroom', 'living_room', 'dining'];
    let needShrink = -gap;
    for (const type of shrinkPriority) {
      const candidates = rooms.filter(r => r.type === type && r.targetArea > r.minArea);
      candidates.forEach(r => {
        const headroom = r.targetArea - r.minArea;
        const sub = Math.min(headroom, needShrink / candidates.length);
        r.targetArea -= sub;
        needShrink -= sub;
      });
      if (needShrink <= 0.1) break;
    }
  }
  return rooms;
}
```

## 6.6 卫生间独立动线检查伪代码（v3 新增）
```javascript
function checkBathroomAccess(plan, kitchenId, bathroomId) {
  // BFS 从客厅出发，检查到卫生间的最短路径
  const living = plan.rooms.find(r => r.type === 'living_room');
  const bath = plan.rooms.find(r => r.type === 'bathroom');
  if (!living || !bath) return { valid: false, reason: '缺少客厅或卫生间' };
  
  const queue = [living];
  const visited = new Set([living]);
  const parent = new Map();
  
  while (queue.length) {
    const cur = queue.shift();
    if (cur === bath) break;
    for (const neighbor of getRoomNeighbors(cur, plan.rooms)) {
      if (!visited.has(neighbor)) {
        visited.add(neighbor);
        parent.set(neighbor, cur);
        queue.push(neighbor);
      }
    }
  }
  
  if (!parent.has(bath)) return { valid: false, reason: '卫生间不可达' };
  
  // 回溯路径，检查是否必须经过厨房
  const path = [];
  let node = bath;
  while (node !== living) {
    path.unshift(node);
    node = parent.get(node);
  }
  
  const passesKitchen = path.some(r => r.type === 'kitchen');
  if (passesKitchen && path.length === 1) {
    return { valid: false, reason: '卫生间只能通过厨房进入', violation: 'R6a' };
  }
  return { valid: true, path };
}
```

# 八、3D联动规则（2D→3D一致性）

1. **轮廓继承**：每层2D外墙轮廓直接作为3D体块基底，层高按楼层设定
2. **楼梯间贯通**：楼梯间在3D中为竖向通高体块（或逐层叠加），内部生成楼梯踏步几何
3. **露台/阳台**：2D中标注的露台区域，在3D中生成底板+栏杆（高1.1m），不生成顶板
4. **屋顶生成**：根据屋顶类型（双坡/四坡/平屋顶）和顶层轮廓生成屋面，考虑出檐
5. **门窗映射**：2D中的门窗位置映射到3D墙面，生成窗洞/门洞
6. **退台处理**：若三层向内退台，3D中相应墙体退进，形成露台或屋面平台

# 十一、与专业级平面图生成器的差距分析（v3 新增）

## 11.1 为什么 ArchyBase / Maket.ai / Finch3D 生成效果好？

你截图中的 ArchyBase 使用的是**数据驱动 + 深度学习**路线，其核心技术栈与当前规则引擎有本质区别：

| 维度 | 当前规则引擎 | 专业 AI 生成器（ArchyBase 类） |
|------|-------------|------------------------------|
| **核心算法** | 矩形分割 + 硬约束检查 | GAN / 图神经网络（HouseGAN、Graph2Plan） |
| **训练数据** | 无，纯规则推导 | 8万+ 真实户型图（RPLAN、LIFULL 数据集） |
| **布局策略** | 固定模板（前带/后带） | 从气泡图（Bubble Diagram）学习最优邻接 |
| **墙线表达** | 简单矩形填充 | 真实墙线+门扇开启弧+家具符号+尺寸标注线 |
| **输出格式** | SVG 方块 + 文字 | 矢量 CAD / DXF / 可编辑 BIM |
| **回溯能力** | 弱（仅输出警告） | 强（MCTS/进化算法自动搜索数千方案） |

## 11.2 如果要达到类似效果，推荐的技术路径

### 路径A：集成现成开源模型（推荐，工作量最小）
- **使用 HouseGAN++ 或 Graph2Plan 的预训练模型**：
  - 这些模型接受"房间数量+邻接图"作为输入，输出高质量户型 mask。
  - 可通过 Python 后端（Flask/FastAPI）调用模型，前端继续用 Three.js 做 3D 预览。
  - **缺点**：需要 GPU 环境（哪怕是 CPU 推理也很慢），模型对农房特殊场景（堂屋、神龛）缺乏训练。
- **使用蒙特卡洛树搜索（MCTS）优化当前引擎**：
  - 不需要神经网络，用 MCTS 在布局空间中搜索满足所有硬约束的最优解。
  - 参考论文：《Addressing adjacency constraints in rectangular floor plans using Monte-Carlo Tree Search》

### 路径B：增强当前规则引擎（无需 AI，可达"够用"水平）
如果你希望保持纯前端、零后端依赖，可以通过以下改进大幅提升质感：

1. **墙线化渲染（最重要）**：
   - 当前 SVG 是直接用矩形填充房间，真实户型图是画墙线（双线或粗线）。
   - 建议改为：先画外墙轮廓线（粗线），再在房间之间画内墙分隔线，最后在墙线上开门洞缺口。
2. **增加家具符号**：
   - 客厅：L型沙发轮廓
   - 卧室：床（1.5m×2.0m 矩形）+ 床头柜
   - 厨房：L型灶台轮廓
   - 卫生间：马桶+淋浴区挡水条
   - 这些符号不需要精确碰撞检测，按比例放在房间中心或贴边即可，极大提升专业感。
3. **增加尺寸标注线**：
   - 在房间之间引出尺寸线（类似 ArchyBase 截图中的 5.0m、3.6m 标注）。
   - SVG 中可用 `<line>` + `<text>` 实现，计算房间边界坐标即可。
4. **门扇开启弧**：
   - 当前只有门洞，建议在平面图中增加 90° 弧线表示门扇开启方向（ArchyBase 截图中有）。
5. **约束求解器（可选）**：
   - 引入一个轻量级 JavaScript 约束求解库，如 **`cassowary.js`**（用于线性约束求解）或 **`webcola`**（用于力导向图布局）。
   - 用 `cassowary.js` 可以精确表达"房间面积 ≥ 最小值"、"楼梯间宽度 = 2.4m" 等约束，自动求解满足所有约束的坐标。

### 路径C：混合方案（中长期推荐）
- 前端保持当前参数面板和 Three.js 预览。
- 增加一个可选的"AI 生成模式"按钮，调用部署在服务器上的 Python 服务（HouseGAN / 自研 MCTS）。
- 本地模式继续使用规则引擎，保证离线可用；AI 模式用于生成更高质量的候选方案。

## 11.3 缺少的"插件"清单

如果你问"我是否缺什么插件"，答案是：

| 你缺的 | 说明 | 获取方式 |
|--------|------|---------|
| **真实户型数据集** | RPLAN 等数据集包含 8 万张真实户型，是训练/验证的核心 | https://github.com/SeanChenxy/RPLAN （需申请） |
| **约束求解器** | `cassowary.js` 或 `webcola` 可以帮助自动求解房间坐标，避免手动写死分割逻辑 | npm install cassowary 或 npm install webcola |
| **家具符号库** | 一套标准化的 SVG 家具图标（沙发、床、马桶、灶台） | 可自行绘制简单几何 SVG，或用 `floorplan.js` 的符号集 |
| **矢量墙线渲染** | 从"色块填充"升级到"墙线分隔"的渲染逻辑 | 需要重写 `renderPlan` 函数，改画墙线而非填充方块 |

> **结论**：在纯前端、无 AI 模型的前提下，通过"墙线化渲染 + 家具符号 + 尺寸标注 + R6a 动线修复 + R4a 自适应填充"，当前生成器可以达到"农房设计参考图"级别的效果，足以满足村庄规划需求。若要达到 ArchyBase 的"专业施工图草稿"级别，则需要引入后端 AI 模型或 MCTS 优化引擎。
